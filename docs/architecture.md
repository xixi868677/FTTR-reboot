## 测试执行流程

```
测试人员触发Jenkins Job
        │
        ▼
Jenkins通过SSH连接测试人员PC (VPN通道)
        │
        ▼
测试PC: git pull 拉取最新代码
        │
        ▼
Robot Framework 执行测试 (循环N轮)
        │
        ├──→ [步骤1] 初始化 - 连接继电器, 获取基准dump.txt
        ├──→ [步骤2] 继电器断电
        ├──→ [步骤3] 验证设备掉电
        ├──→ [步骤4] 断电保持等待
        ├──→ [步骤5] 继电器上电
        ├──→ [步骤6] 轮询等待设备恢复
        ├──→ [步骤7] telnet获取dump.txt
        ├──→ [步骤8] 验证dump.txt内容
        └──→ [步骤9] 重复步骤2~8 (共N次)
        │
        ▼
测试完成后自动执行:
        │
        ├──→ 1. 解析output.xml, 按步骤同步结果到禅道用例853
        │        (步骤级PASS/FAIL/BLOCKED, 失败原因)
        │
        ├──→ 2. 发送测试结果到企业微信群
        │        (通过/失败, 失败步骤, 失败原因)
        │
        └──→ 3. SCP拷贝output.xml回Jenkins展示报告
```

## 技术栈

| 组件 | 技术 | 说明 |
|------|------|------|
| 测试框架 | Robot Framework 7.4.2 + Python 3.10 | 关键字驱动测试 |
| 继电器控制 | pyserial (COM3) | USB转串口, hex指令控制 |
| 设备交互 | telnetlib | 登录FTTR主设备获取dump.txt |
| 禅道同步 | requests + 禅道API v1 | 逐步骤提交PASS/FAIL/BLOCKED |
| 通知推送 | 企业微信Webhook | markdown格式群消息 |
| CI/CD | Jenkins + SSH + VPN | 远程触发+报告回收 |
| 代码管理 | GitHub | 版本控制, 代码仓库 |

## AI辅助开发过程

```
1. 测试人员用自然语言描述测试需求:
   "FTTR设备断电重启5次,每次验证设备恢复正常,结果同步禅道"

2. Claude生成完整项目结构:
   - Robot Framework测试用例 (8个步骤的循环测试)
   - Python库: 继电器控制、设备检查、禅道同步、微信通知
   - 配置文件、Jenkins脚本

3. 迭代优化:
   - 修复telnet连接超时 → 添加重试机制(10次,15s间隔)
   - 修复禅道API路径 → /testcases/ 不是 /cases/
   - 修复步骤结果映射 → 逐步骤PASS/FAIL/BLOCKED
   - 修复步骤号提取 → len("[步骤")=3不是4
   - Jenkins集成 → SSH+VPN+企业微信通知
```
